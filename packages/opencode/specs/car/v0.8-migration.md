# CAR v0.7 → v0.9 migration

Status: shipped against `car-runtime@0.9.0` (released 2026-05-11). v0.8.0
made the FFI a thin daemon client; v0.8.1/0.8.2 hotfixed install issues;
v0.9.0 added multi-tenant scoping, daemon-routed reasoning, and the two
FFI exports that were blocking us from v0.8 (`registerToolHandler`,
`registerModel`).

## The defining shift

The NAPI binding no longer hosts an in-process engine. Every method on
`CarRuntime` (and the standalone callback-bearing functions) proxies to
the singleton `car-server` daemon over WebSocket JSON-RPC. `car-server`
must be running before the first daemon-routed call; the binding
auto-reads the per-launch auth token from
`~/Library/Application Support/ai.parslee.car/auth-token` (macOS) /
`$XDG_RUNTIME_DIR/ai.parslee.car/auth-token` (Linux) on connect.
`CAR_FFI_MODE` is gone — daemon is the only mode.

## What changed in opencode

| Call site | v0.7 | v0.9 |
| --- | --- | --- |
| `index.ts` (`executeAction`) | `executeProposal(rt, json, dispatcher)` | `registerToolHandler` once globally + `rt.submitProposal(json)`; per-tool serialization compensates for the napi shim dropping `action_id` from the JS callback (only `{tool, params}` reaches JS). |
| `inference-bridge.ts` (dispatch) | `inferStream(rt, requestJson, () => {})` | `rt.inferTrackedWithRequest(requestJson)` — same daemon-routed delegated path; the runner still streams via the in-memory JS queue. |
| `inference-bridge.ts` (correlation) | `_opencode_call_id` smuggled in `requestJson` | Smuggled via `prompt` field. `GenerateRequest` is a typed Rust struct so unknown fields are dropped on serde round-trip; `prompt` survives and the delegated runner doesn't use it (delegated models own their own prompt construction via `ctx.params`). |
| `inference-bridge.ts` (model) | `rt.registerModel(...)` (sync) | `await rt.registerModel(...)` — now a daemon-proxied `Promise<string>`. Visibility limitation: the model only becomes routable on next daemon boot; we register before first inference, so the first opencode-spawned daemon picks it up at startup. |
| `bin/opencode`, root `package.json` `dev`, `src/car/env.ts` | `CAR_FFI_MODE=embedded` setter | Removed. The env.ts tombstone module is gone. |
| Daemon lifecycle | — | New `src/car/daemon.ts` — probes `ws://127.0.0.1:9100` (or `CAR_DAEMON_URL`); if unreachable, spawns the `car-server-<platform>` binary from `node_modules/car-runtime/` detached, polls for readiness. The macOS menubar app and Linux systemd unit, where present, are picked up transparently. |

## What stays unchanged

`registerInferenceRunner` and `inferenceRunnerEmitEvent` remain
process-wide standalone functions in v0.8+ and the JS-side runner body
is unchanged. `CarRuntime` methods we already used (`addFact`,
`factCount`, `buildContext`, `ingestSkill`, `findSkill`, `listSkills`,
`verifyProposal`, `registerTool`, `registerToolSchema`,
`loadMemory`/`persistMemory`) keep their existing signatures and proxy
to the daemon transparently.

## Behaviour vs v0.7

1. **Concurrent same-tool `executeAction` parity restored.** v0.9.0's
   napi `tools.execute` bridge dropped the daemon's `action_id` from
   the JS payload, breaking process-wide handler routing under
   concurrency. We patched it upstream — both
   `submit_proposal`'s persistent bridge and `execute_proposal`'s
   per-call bridge now include `action_id` in the `{tool, params,
   action_id}` callJson (see `car-rs/crates/car-ffi-napi/src/lib.rs`
   and CHANGELOG `Unreleased`). opencode's process-wide
   `registerToolHandler` routes by `action_id` straight off the
   payload — no per-tool serialization.
2. **Inference correlation: `prompt` instead of `_opencode_call_id`.**
   `GenerateRequest` is a typed Rust struct, so v0.7's smuggled
   `_opencode_call_id` field is dropped on serde round-trip. The
   delegated path ignores `prompt` for prompt construction (the
   runner uses `ctx.params` for `streamText`), so we carry the
   correlation id there. Multiple concurrent `runInference` calls
   work because each gets a fresh `randomUUID()`.
3. **`registerModel` hot-update — sidestepped, not fixed.** v0.9.0's
   `models.register` handler still has the phase-1 limitation: it
   writes `~/.car/models.json` but doesn't refresh the daemon's
   in-memory `UnifiedRegistry` (hot-update touches ~90
   `unified_registry` call sites — tracked as upstream follow-up).
   opencode sidesteps it: `daemon.ts::ensureCarServer()` writes the
   `opencode-delegated` schema to `~/.car/models.json` BEFORE
   spawning the daemon, so the daemon's `load_user_config` picks it
   up at boot. We no longer call `rt.registerModel` at all — it
   would only duplicate the file write.

   **Caveat**: when an externally-managed daemon (macOS menubar app,
   systemd unit) was running before opencode's first invocation and
   doesn't have `opencode-delegated` in `~/.car/models.json`, the
   first inference call will fail with "model not found." The user
   needs to restart the daemon after opencode's first run wrote the
   file. Once `UnifiedRegistry` gains hot-update, this caveat
   disappears.

## What v0.9 newly enables (not adopted yet)

- **Daemon-owned per-agent persistent memgine** (#170): connection-level
  agent identity via `session.auth { agent_id, token }` attaches the
  WS session to `~/.car/memory/agents/<id>.json`. Could replace
  opencode's per-project `~/.local/share/opencode/car/<projectID>.json`
  snapshot dance.
- **MCP server endpoint at `/mcp`**: external agents (Claude Code,
  Codex, Gemini) can share opencode's memgine through CAR's MCP.
- **External agent dispatch** (`agentsInvokeExternal`): first-class
  adapters for Claude Code / Codex / Gemini CLIs as subagents.
- **Per-session policy scoping** (`session.open` / `session.close` +
  `policy.register { session_id }`): declarative permission policies
  for opencode sessions.
- **Multi-tenant scoping** (car#187 phase 3): `RuntimeScope`
  `{ tenantId, callerId, claims }` on `proposal.submit`. Not relevant
  for single-tenant opencode but available if we surface multi-project
  isolation later.
