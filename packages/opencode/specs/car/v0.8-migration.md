# CAR v0.8 migration plan

Status: planning. opencode tracks released CAR versions; this spec captures what
moving from v0.7.0 → v0.8.x will require, so we can ship the migration in one
focused PR rather than discovering it commit-by-commit.

Sources: `../car/CHANGELOG.md` (local v0.8.0 / v0.8.1 / unreleased entries),
`https://unpkg.com/car-runtime@0.8.1/index.d.ts`, `../car/docs/websocket-protocol.md`.

## The defining shift

v0.8 makes the FFI bindings **thin daemon clients**. NAPI and PyO3 no longer
host an in-process engine. Every non-callback method proxies to the singleton
`car-server` daemon over WebSocket. `CAR_FFI_MODE=embedded` is retired.

Quoting `car-runtime@0.8.1/index.d.ts` (v0.8.0 — daemon-only):
> Start `car-server` before using the bindings — on macOS the SwiftUI menubar
> app launches it for you; on Linux start it manually or via systemd.

## What breaks for opencode

Three call sites in `packages/opencode/src/car/` use FFI surfaces that throw in
v0.8. The functions still exist in the `.d.ts` but the Rust shims are gone
(`car-ffi-napi` has no `register_inference_runner` symbol in v0.8.1).

| Site | v0.7 call | v0.8 status | Replacement |
| --- | --- | --- | --- |
| `index.ts:213` (`executeAction`) | `executeProposal(rt, proposalJson, dispatcher)` | throws | Daemon WS: `proposal.submit` JSON-RPC + register a `tools.execute` handler on the connection. The dispatcher closure becomes the handler body. |
| `inference-bridge.ts:96` (runner registration) | `registerInferenceRunner(fn)` | throws | Daemon WS: `inference.register_runner` JSON-RPC. The FFI stub for the same name is gone. |
| `inference-bridge.ts:118` (event emit) | `inferenceRunnerEmitEvent(callId, eventJson)` | throws | Daemon WS: `inference.runner.event` JSON-RPC for each chunk; `inference.runner.complete` / `inference.runner.fail` at the end. |
| `inference-bridge.ts:172` (inference dispatch) | `inferStream(rt, requestJson, () => {})` | throws | Daemon WS: `infer_stream` JSON-RPC. Server pushes `inference.stream.event` notifications matched by request id. |
| `inference-bridge.ts:154` (`ensureModelRegistered`) | `rt.registerModel(schemaJson)` | throws | Daemon owns `~/.car/models.json`. Either write a config file the daemon picks up at start, or send a `models.register` JSON-RPC (TBD — not currently in `models.*` namespace). |
| `index.ts:102` (CAR_FFI_MODE) | `process.env.CAR_FFI_MODE = "embedded"` | tombstone | Drop. Daemon is the only mode. Keep `CAR_DAEMON_URL` override and `CAR_AUTH_TOKEN` reading. |

## What stays

The class methods on `CarRuntime` that we already use (`addFact`, `factCount`,
`buildContext`, `ingestSkill`, `findSkill`, `listSkills`, `verifyProposal`,
`loadMemory`, `persistMemory`, `registerTool`, `registerToolSchema`) keep their
existing signatures. They proxy to the daemon transparently. opencode's
`src/car/index.ts` Service interface is unchanged for these.

`registerToolSchema` is unaffected — schemas are sent over WS and validated
daemon-side before `tools.execute` callbacks fire.

## What we get

Useful surfaces v0.8 adds that opencode could adopt opportunistically (none are
required by the migration itself):

- **Daemon-owned per-agent persistent memgine** (unreleased, #170). Connections
  authenticate with an `agent_id`; daemon attaches them to
  `~/.car/memory/agents/<id>.json` keyed on that id, persists across daemon
  restarts. Could replace our per-project `~/.local/share/opencode/car/<projectID>.json`
  snapshot dance. Adoption: pick a stable `agent_id` derived from the project id;
  call `session.auth { agent_id, token }`; let CAR's memgine survive shutdown
  instead of `persistMemory` / `loadMemory` round-trips.
- **MCP server endpoint at `/mcp`** (default `127.0.0.1:9102`). The daemon
  exposes CAR's tools (`memory_add_fact`, `memory_query`, `verify`, `skill_*`)
  as MCP. opencode itself doesn't need this — we already wire tool execution
  through CAR — but it's the seam external agents (Claude Code, Codex, Gemini)
  use to share opencode's memgine.
- **External agent dispatch via `agents.invoke_external`**. Lets opencode
  invoke Claude Code / Codex / Gemini CLIs as agents inside the same CAR
  memgine and policy pipeline. Maps to the existing "Coming next" multi-agent
  item but with first-class adapters instead of `runSwarm` callbacks.
- **Per-session policy scoping** (`session.open` / `session.close` +
  `policy.register { session_id }`). The README's "Coming next" mentioned
  declarative permission policies as deferred; v0.8 makes per-session scoping
  cheap enough to wire if we want to.

## Migration tasks

Ordered so each step can ship and typecheck independently.

### 1. Daemon discovery and lifecycle

- Detect `car-server` on `$PATH`. Print a helpful message if missing — link to
  install instructions.
- Read `CAR_DAEMON_URL` (default `ws://127.0.0.1:9100`).
- Read `CAR_AUTH_TOKEN` from env, or from
  `~/Library/Application Support/ai.parslee.car/auth-token` (macOS) /
  `$XDG_RUNTIME_DIR/ai.parslee.car/auth-token` (Linux).
- Decide whether opencode spawns the daemon itself, requires the user to run
  it, or both. Spawn-on-demand is friendlier for `bun run dev` but adds a
  parent/child supervision concern. Recommended: spawn-if-missing with
  `child_process.spawn(carServerPath, [], { detached: true })` and a 2s
  probe-and-wait loop; fall through to a clear error if spawn fails.

The CarRuntime constructor itself connects lazily, so opencode doesn't need to
do anything special on the FFI side beyond making sure the daemon is reachable
before the first daemon-routed method call.

### 2. Tool execution: `executeProposal` → WS

Today `executeAction` (`src/car/index.ts:174-229`) does this in one shot via
`executeProposal`. v0.8 splits it across two methods on different sessions:

- `proposal.submit { proposal }` — submit the proposal, get a result.
- A persistent `tools.execute` handler installed on this WS connection,
  invoked when the daemon needs the JS-side tool implementation to run.

Implementation outline:

- Subclass-or-wrap the CarRuntime instance with a `DaemonClient` (`car-ffi-common`
  Rust API — not currently exposed via NAPI; we'll need either a WS client of
  our own or, ideally, an NAPI export from CAR).
- Register one global `tools.execute` handler keyed by `(sessionID, callID)`
  that dispatches to the same closure `executeAction`'s `dispatcher` currently
  does.
- Replace the `executeProposal` call with a `proposal.submit` JSON-RPC.

Open question: does v0.8 expose `register_handler` (the Rust DaemonClient
method) over NAPI? If not, opencode needs its own minimal WebSocket client. The
v0.8.1 d.ts doesn't show it, but `inference.register_runner` exists, so the
FFI clearly has SOME way to install callbacks — likely a not-yet-documented
NAPI method we'll need to dig out of `car-ffi-napi/src/`.

### 3. Inference: `registerInferenceRunner` → WS

`src/car/inference-bridge.ts` becomes a thin WebSocket client of the daemon's
`inference.*` namespace. The side-channel design stays the same — runner runs
`streamText`, fans rich AI-SDK chunks to opencode's processor via a JS queue,
fans simplified events to CAR via what's now `inference.runner.event` JSON-RPC.

- Replace `registerInferenceRunner(fn)` with one `inference.register_runner`
  call at startup, plus a notification handler for `inference.runner.invoke`
  that routes invocations into the same runner body.
- Replace each `inferenceRunnerEmitEvent(callId, eventJson)` with a
  `inference.runner.event { call_id, event }` JSON-RPC call.
- On runner completion, send `inference.runner.complete { call_id, result }`
  (success) or `inference.runner.fail { call_id, error }` (failure).
- Replace `inferStream(rt, requestJson, () => {})` with `infer_stream` JSON-RPC.
  Capture the original RPC id; correlate `inference.stream.event` notifications
  by that id back into the side-channel.

### 4. Model registration

The current bridge calls `rt.registerModel(...)` to declare `opencode-delegated`
as a delegated source. In v0.8 the FFI method is gone. Three options:

- **Write `~/.car/models.json` ourselves before daemon start.** Daemon picks up
  the file at boot. Brittle — opencode has to know daemon's filesystem
  conventions and any time CAR changes the schema we silently break.
- **Send a `models.register` JSON-RPC** at opencode startup. Not currently in
  the protocol (`websocket-protocol.md` `models.*` section: `list`,
  `list_unified`, `search`, `upgrades`, `pull`, `install`, `route`, `stats` —
  no `register`).
- **Open a CAR issue** asking for `models.register` JSON-RPC parity with the
  v0.7 FFI method (currently retired without a daemon-side replacement).

Best path: open the issue. Until then, write the file directly.

### 5. Drop CAR_FFI_MODE plumbing

- Delete `packages/opencode/src/car/env.ts` and the `import "./env"` lines.
- Drop `CAR_FFI_MODE=embedded` from `package.json` `dev` script and
  `packages/opencode/bin/opencode`.
- Keep the daemon-discovery work from step 1; `CAR_DAEMON_URL` and
  `CAR_AUTH_TOKEN` are the relevant env vars now.

### 6. Test surface

The `packages/opencode/test/installation/installation.test.ts` is unaffected
(no CAR coupling). The CAR layer's existing in-memory tests stay; the
end-to-end inference path now needs a daemon to test against. Options:

- Skip end-to-end inference tests when daemon is unreachable (most paths).
- Spawn a daemon in a Bun test setup hook (slow; only run in CI).

## Risks

- **registerHandler / NAPI gap.** If v0.8 doesn't expose a NAPI export for
  installing tool-execute handlers, opencode either ships its own WS client
  (~300 LOC, includes the auth-token / reconnect handshake from
  `docs/websocket-protocol.md`) or waits for CAR to expose one.
- **models.register parity gap.** Without it we either touch `~/.car/models.json`
  directly (fragile) or block the migration on a CAR feature.
- **Daemon lifecycle.** `opencode run` is a short-lived CLI; auto-spawning the
  daemon on first invocation and letting it persist is the only way the
  singleton-per-host contract actually pays off. Need to decide whether
  opencode owns daemon teardown (it shouldn't if other consumers might be
  attached) or relies on the macOS menubar app + systemd unit on Linux.

## Timeline

- v0.8.0 / v0.8.1 are released as of 2026-05-11.
- v0.8.2 hotfix prepared (macOS npm dlopen fix), pending github release.
- Unreleased entries cover #170-#175, mostly daemon-side polish and per-agent
  memgine + supervisor child-env work. None of these block the migration; we
  can ship against v0.8.1 the moment the macOS npm install is stable.

Estimate the full migration at 1-2 working days once `models.register` and
the NAPI `register_handler` gaps are closed. Without those, it's larger
because opencode has to maintain a WS client of its own.
